Cloning vs. Forking

Clone a repository

Cloning is good when you want a copy of the original project on your local machine but you don't intend to contribute back to the original project.

The other scenario, this is our case, is that you are invited as a collaborator.

When you clone a repository, you are not linked with the remote. You can't create pull requests back to the original remote unless you are invited as a collaborator and can contribute back.

Fork a repository

You can fork (create a local copy) of any publicly visible repository on GitHub.

A fork is always linked to the parent repository and is always public.

It is good for open-source projects collaboration when you want to bring in new ideas to the project and create pull requests.
